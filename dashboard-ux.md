# Contra Dashboard UX 研究设计

> 基于 [Dashboard PRD v6](./dashboard-prd.md) 的 UX 研究文档  
> 目标用户：Web3 × AI 咨询协议的 4 种身份持有者  

---

## 1. 用户场景分析

Contra Dashboard 服务 4 种身份，每种身份在 Dashboard 上的**核心任务路径**不同。  
下面按"首次进入 → 核心操作 → 完成目标 → 离开"梳理每种身份的旅程。

### 1.1 场景 A：Partner 身份持有者（质押治理型）

| 阶段 | 描述 |
|------|------|
| **入口意图** | 连接钱包，确认身份 NFT，查看质押收益和投票权重 |
| **关键任务 1** | 在收益区域快速确认 APR 和待提取金额，决定是否 Claim |
| **关键任务 2** | 切换到 Partner Tab，查看当前质押额和配额进度 |
| **关键任务 3** | 进入治理投票区，使用高权重影响提案方向 |
| **退出状态** | 完成 Claim + Vote 后离开，预期下次周期性回访 |
| **情绪曲线** | 初入确认身份时的安心 → 看到收益时的满足 → 投票时的参与感 |

**设计重点**：Partner 用户对 Dashboard 的核心诉求是"控制感"——一眼看到自己的质押健康度和治理影响力。收益环图 + 配额进度条是满足这一诉求的关键组件。

### 1.2 场景 B：User 身份持有者（消费服务型）

| 阶段 | 描述 |
|------|------|
| **入口意图** | 快速确认身份状态，浏览可用的 Expert 服务 |
| **关键任务 1** | 确认 NFT 持有状态，查看权益（服务折扣、优先匹配等） |
| **关键任务 2** | 若未铸造，评估铸造条件和质押额，决定是否 Mint |
| **关键任务 3** | 查看活动记录中的 Staking/Claim 历史 |
| **退出状态** | 身份确认后跳转至服务市场或咨询匹配 |
| **情绪曲线** | 好奇 → 条件评估时的计算感 → 权益确认后的满足或未铸造时的动机 |

**设计重点**：User 身份需要清晰的"权益可视化"——2×2 grid 需一目了然展示"你有哪些特权"。Mint 按钮的 CTA 要强化铸造后的价值获得感。

### 1.3 场景 C：Expert 身份持有者（声誉经济型）

| 阶段 | 描述 |
|------|------|
| **入口意图** | 查看声誉分数变化、任务完成情况和收益 |
| **关键任务 1** | 铸造后查看专属数据面板：声誉值、完成任务数、累计收益 |
| **关键任务 2** | 监控配额使用进度（Expert 可能有服务配额） |
| **关键任务 3** | 参与治理投票（Expert 通常有较高权重） |
| **退出状态** | 确认声誉和任务状态后，转向接单或交付 |
| **情绪曲线** | 职业型冷静 → 声誉增长的成就感 → 数据驱动的决策 |

**设计重点**：Expert 需要"职业面板"的体感——Expert Tab 铸造后的专属字段（声誉/任务/收益）应设计为类似 LinkedIn Dashboard 的数据卡片风格，给专家以专业认可感。

### 1.4 场景 D：KOL 身份持有者（影响力增长型）

| 阶段 | 描述 |
|------|------|
| **入口意图** | 查看推荐数据、佣金收入和影响力指标 |
| **关键任务 1** | 铸造后查看专属数据：推荐人数、佣金总额、影响力分数 |
| **关键任务 2** | 从活动记录追踪推荐转化（Stake/Claim 类型） |
| **关键任务 3** | 投票时利用自身影响力（高权重） |
| **退出状态** | 数据确认后可能截屏分享或规划推广策略 |
| **情绪曲线** | 数据期待 → 增长可视化的兴奋 → 分享和传播动机 |

**设计重点**：KOL 身份的核心心理需求是"增长可视化"——推荐数、佣金、影响力三指标需要醒目展示，建议在已铸造面板中加入微动效（数字滚动、进度增长），强化增长感知。

### 1.5 四身份路径对比

| 维度 | Partner | User | Expert | KOL |
|------|---------|------|--------|-----|
| 核心需求 | 治理控制 | 服务消费 | 声誉建设 | 影响力变现 |
| 首要关注区域 | 收益环 + 投票 | NFT权益 + Mint | Expert数据面板 | KOL数据面板 |
| 关键操作 | Claim / Vote | Mint | 查看声誉 | 追踪推荐 |
| 回访频率 | 周期性（治理周期） | 按需（服务消费） | 日常（接单交付） | 定期（监控增长） |
| 决策驱动 | 数据 + 提案 | 权益 + 条件 | 声誉数据 | 增长指标 |

**设计原则**：
- **F 型扫描**：用户视线先从左上到右横扫 Topbar + Stats Bar，然后沿左侧下扫 Earnings Ring → Tab，最后扫描 Governance 和 Activity
- **渐进披露**：高频信息（收益环、NFT状态）优先暴露；低频信息（活动历史）使用表格承载
- **色彩编码**：每身份对应一个主色，贯穿 Tab、NFT 卡片、投票权重等元素

## 3. 交互设计建议

### 3.1 Tab 切换交互

**当前方案**：click → class toggle → fade-in transition

**增强建议**：

| 交互要素 | 建议 | 理由 |
|----------|------|------|
| **触发方式** | 保留 click，增加左右滑动手势（移动端） | 移动端横向 Tab 栏需要 swipe 支持 |
| **过渡动画** | `opacity 150ms ease-out` + `transform: translateY(4px) → 0`（微上浮） | 纯 fade 偏平，加微位移增加层次感 |
| **激活态反馈** | Tab 文字颜色从灰→身份色 + 底部 3px 身份色下划线 + 轻微 scale(1.05) | 多通道反馈（色+形+大小）确保可感知 |
| **内容切换** | 面板使用 CSS `@starting-style` + `transition` 实现 SPA 式切换 | 避免硬切，提升品质感 |
| **首屏默认** | 默认展示用户当前持有 NFT 的 Tab；若持多个，优先展示最高权益身份 | 减少用户切换成本 |
| **空状态 Tab** | 未铸造的身份 Tab 仍然可点击，内容区显示"未铸造"引导状态 | 让用户了解其他身份，可能促进升级 |

**代码骨架建议**（React/Vue 通用模式）：
```
Tab 切换组件结构：
TabBar (role="tablist")
  ├── Tab[0] (role="tab", aria-selected, aria-controls="panel-0")
  ├── Tab[1]
  ├── Tab[2]
  └── Tab[3]
TabPanels
  ├── Panel[0] (role="tabpanel", aria-labelledby="tab-0")
  └── ...
```

**关键 A11Y**：Tab 需实现完整的 ARIA 规范（tablist / tab / tabpanel），支持键盘 ← → 导航。

### 3.2 投票弹窗 UX

**流程分解**：
```
点击 Vote 按钮
  → Modal 从 Vote 按钮位置 scale 展开（锚点动画，150ms）
  → 显示提案标题 + 描述（可滚动）
  → 显示三个单选卡片（For / Against / Abstain）
  → 选中态：卡片高亮 + 边框 + 勾选 icon
  → 点击 Confirm
    → 按钮变为 loading spinner "Confirming..."
    → 模拟 1.2s 交易（+ 进度条或 3 步伪交易流）
    → 成功：Modal 内容切换为 Success View
      → ✅ 动画 + "Vote Submitted" + TxHash
      → 关闭 → 提案列表自动更新
    → 失败：Error View（重试按钮）
  → 或关闭 Modal（点击遮罩 / × / Esc）
```

**关键交互细节**：

| 细节 | 建议 | 避免 |
|------|------|------|
| **弹窗定位** | 居中弹出，移动端变为底部 Sheet（Bottom Sheet） | 移动端也用居中 Modal 会遮挡内容且键盘问题 |
| **选项设计** | 三张水平卡片，带图标（👍 For / 👎 Against / ⏸️ Abstain） | 纯文字 radio button，点击区域太小 |
| **确认按钮** | 未选择时 disabled 且显示提示"Select a position to continue" | 永远的 disabled 按钮让用户困惑 |
| **模拟交易** | 3 步伪交易流：Sign → Confirm → Broadcast，每步 400ms | 单一步骤过短（<500ms）显得不可信 |
| **撤销设计** | 提交后 Modal 内可展示"投票成功"3 秒，但**无撤销功能**（模拟链上不可逆） | 给用户可以撤回的错觉 |
| **关闭行为** | 未提交关闭 = 丢弃选择；已提交关闭 = 更新列表 | 未提交时需二次确认关闭 |

**移动端 Bottom Sheet 结构**：
```
┌──────────────────────┐
│  ━━━━ (拖拽手柄)      │
│  提案 #3: 标题        │
│  ───────────────────  │
│  描述内容（可滚动）     │
│  ───────────────────  │
│  [👍 For] [👎 Against] │  ← 三张点击卡片
│  [⏸️ Abstain]         │
│  ───────────────────  │
│  [  Confirm Vote  ]   │  ← 全宽按钮，未选时灰色
└──────────────────────┘
```

### 3.3 收益展示与认知负荷管理

**问题分析**：收益区域同时包含环形进度条、APR%、三列数字卡片，容易造成信息过载。

**建议方案**：

| 策略 | 实现方式 |
|------|----------|
| **视觉焦点层级** | 环形图作为第一焦点（240px，最大组件）；三列卡片作为第二焦点（较小字号） |
| **数字格式化** | `$12,450` 而非 `12450`；大数字用 `3.24K` 缩写（移动端）；小数位统一 2 位 |
| **色彩语义** | APR 数字用渐变色（绿→紫），与环形图色彩呼应；Pending 用琥珀色暗示"等待中" |
| **数据解释** | 每个卡片旁放 `ⓘ` icon，hover/tap 显示 Tooltip 解释计算方式 |
| **加载顺序** | 环形图先加载（最重要），卡片延迟 100ms 依次出现（stagger animation），避免一次性涌入 |
| **空值处理** | 新用户 APY 显示 `--` + "Stake to earn" 引导文案，而非 0.00% |

**认知负荷计算**：
```
Earnings Section 信息量：
- 环形图：已提取 / 总额（2 个数字） + APR%（1 个数字）
- 三列卡片：Monthly / Total / Pending（3 个数字）
- 合计：6 个数字 → 在 Miller's Law（7±2）范围内 ✅

建议：确保同一时刻只有 4-6 个数字可见，其余通过 Tooltip/展开渐进披露。
```

### 3.4 Mint 交互流

```
铸造前 Tab → 点击 Mint 按钮
  → 按钮显示 "Confirming..." + spinner
  → 模拟交易（1.5s）
  → 成功：
    → Tab 内容区从铸造面板 fade-out + slide-up
    → 已铸造面板 fade-in + slide-up-from-bottom
    → 顶部 Stats Bar 的 NFT Count 数字 +1（带动画）
    → Mint 按钮变为 disabled "Minted ✅"
  → 失败：Toast 错误提示，按钮恢复可点
```

**关键细节**：Mint 成功后 Tab 内容区需要**平滑过渡**到已铸造面板，不应出现白屏/闪烁。建议使用 React Transition Group 或 Vue `<Transition>` 的 mode="out-in"。

## 4. 可用性考虑

### 4.1 移动端适配

| 断点 | 布局策略 |
|------|----------|
| **≥ 1024px** | 完整桌面布局，环形图 240px，Tab 全展开 |
| **768-1023px** | 中等屏幕，环形图缩小至 180px，三列卡片变两列 |
| **< 768px** | 移动端，环形图居上 200px，三列卡片改为横向滑动（horizontal scroll） |
| **< 480px** | 小屏手机，环形图 160px，Tab 横向滚动，投票 Modal 改为 Bottom Sheet |

**移动端关键适配清单**：

- [ ] Topbar：钱包地址截断为 0x12...abcd，移除余额 USD 换算只保留 Token
- [ ] Quick Stats：四卡片改为 2×2 grid（移动端）
- [ ] Earnings Ring：环形图居中，三列卡片变成横向滑动容器（overflow-x: auto + snap）
- [ ] Tab Bar：横向滚动（`overflow-x: auto`），隐藏滚动条但保留滑动
- [ ] NFT 2×2 Grid：移动端改为竖向堆叠
- [ ] 治理投票：提案列表改为卡片式（移动端表格体验差）
- [ ] 活动记录：同理改为卡片式，只显示 Type + Time + Amount，TxHash 折叠
- [ ] 所有 Modal → Bottom Sheet（iOS/Android 原生感）
- [ ] 所有按钮：最小触控区域 44×44px（WCAG 2.1）

### 4.2 一键可达性（One-Tap Accessibility）

**核心操作触达次数**（从 Dashboard 首屏算起）：

| 操作 | 触达步骤 | 目标 |
|------|----------|------|
| 查看收益 | 0（首屏可见） | ≤1 |
| Claim 收益 | 1（点击环图旁 Claim 按钮） | ≤1 |
| 切换身份 Tab | 1（点击 Tab） | ≤1 |
| Mint NFT | 2（切换 Tab → 点击 Mint） | ≤2 |
| 投票 | 2-3（定位提案 → 点击 Vote → 选择 → Confirm） | ≤3 |
| 查看活动历史 | 1（滚动到 Activity Feed） | ≤1 |

**Fitts' Law 优化**：
- Mint 按钮放在 Tab Panel 右下方，靠近手指热区（移动端拇指热区）
- Vote 按钮放在每行右侧，固定位置，支持快速连点
- Claim 按钮靠近环形图底部，视线流自然到达

### 4.3 状态反馈

#### Loading 状态

| 区域 | Loading 策略 |
|------|-------------|
| **Stats Bar** | 骨架屏（Skeleton），4 个色块脉冲动画 |
| **Earnings Ring** | 环形图灰色虚线描边 + 中心 APR 显示 `--` |
| **NFT Tab** | 卡片骨架屏（2 行骨架块 + 1 个大块模拟权益 grid） |
| **Governance** | 提案列表骨架屏（6 行骨架） |
| **Activity** | 表格骨架屏（10 行骨架） |
| **Mint 按钮** | 点击后立即切换为加载态 spinner，1.5s 后回调 |
| **Vote 按钮** | 点击后弹窗内 Confirm 按钮变 spinner |

**骨架屏节奏**：所有骨架屏应在 2-3 秒内依次替换为真实内容（stagger reveal），营造"快速加载"体感。

#### Empty 状态

| 场景 | Empty 设计 |
|------|-----------|
| **无收益**（新用户） | 环形图显示 0% + 中心文案"Stake to start earning" + CTA 按钮 |
| **未持有 NFT** | NFT 区显示默认 Tab + 引导卡片"Choose your identity and mint your first NFT" |
| **无可投提案** | 治理区显示"All proposals have been voted" + 插画 |
| **无活动记录** | 表格区显示"No activity yet" + 插画 |

**Empty 设计原则**：
1. 不空白——始终有引导性文案 + CTA
2. 不指责——用"Start earning"而非"You have no earnings"
3. 有路径——每个 Empty 状态都包含下一步操作入口

#### Error 状态

| 场景 | Error 处理 |
|------|-----------|
| **数据加载失败** | 对应区域显示"Failed to load" + Retry 按钮 + 错误信息 |
| **Mint 失败** | Toast 错误提示（顶部滑入，3s 自动消失）+ 按钮恢复 |
| **Vote 失败** | Modal 内 Error View 替换表单，显示"Transaction failed" + TxHash + Retry |
| **钱包断开** | Topbar 钱包地址变红闪烁 + 全局提示条"Wallet disconnected" |

**Error 设计原则**：
1. 明确告知发生了什么
2. 提供恢复路径（Retry / Reconnect）
3. 不丢失用户已输入数据

#### Success 状态

| 场景 | Success 反馈 |
|------|-------------|
| **Claim 成功** | 环形进度条微动效（15° 正向摆动）+ Toast "Claimed 3240 CONTRA" |
| **Mint 成功** | Tab Panel 内容切换动画 + Stats Bar NFT Count 数字跳动 + Toast |
| **Vote 成功** | Modal 内 Success View + 列表自动更新（已投票行变色） |
| **数据刷新** | 静默更新（不弹 Toast），只有数字变化用细微的颜色闪烁提示 |

**Success 反馈量级**：
- 高频操作（Claim、刷新）：轻量反馈（微动效 + Toast）
- 低频操作（Mint、Vote）：完整反馈（动画 + 内容更新 + Toast）
- 不要过度使用 confetti 爆花等重动效——Web3 用户偏好专业感

### 4.4 动画节奏规范

| 用途 | 时长 | 缓动 | 说明 |
|------|------|------|------|
| Tab 切换 | 150ms | `ease-out` | 快速切换，不拖沓 |
| Modal 进出 | 200ms | `cubic-bezier(0.16,1,0.3,1)` | Apple 风格弹性动画 |
| 数字跳动 | 600ms | `ease-out` | 数字从旧值滚动到新值 |
| 环形进度 | 800ms | `ease-in-out` | 进入视口时动画填充 |
| 骨架屏脉冲 | 1.5s 循环 | `ease-in-out` | 柔和呼吸效果 |
| Toast 滑入 | 300ms | `ease-out` | 顶部滑入 |

**减少动画**：使用 `prefers-reduced-motion` 媒体查询，当用户系统开启"减少动效"时，所有动画时长缩减为 0ms 或替换为瞬间过渡。

## 5. 竞品参考

### 5.1 直接竞品（Web3 Dashboard）

| 产品 | 值得借鉴 | 需避免 |
|------|----------|--------|
| **Aave** | 收益展示极简，环形 APY 居中突出；supply/borrow 切换清晰 | 信息密度过高，新用户认知负荷大 |
| **Compound** | 治理提案列表设计清晰，状态色彩编码（绿 Active/灰 Ended） | 投票流程步骤偏多（4-5 步），不够流畅 |
| **Uniswap** | LP 位置管理面板的 UI 简洁，手续费收益一目了然 | 缺少身份/角色区分，所有用户看同一界面 |
| **ENS** | NFT 域名管理面板的卡片式设计，注册/续费流程清晰 | 数据面板过于简单，缺少收益分析 |
| **MakerDAO** | 治理投票面板成熟，提案详情 + 投票历史分离清晰 | 偏技术化语言，非技术用户有阅读障碍 |

### 5.2 间接竞品（Web2 Dashboard / DeFi）

| 产品 | 值得借鉴 |
|------|----------|
| **Stripe Dashboard** | 收益图表 + 数字卡片的布局最优实践，数据层次清晰 |
| **Notion** | 身份/角色切换的流畅体验，多身份管理不混乱 |
| **1Password** | Tab + 分类管理的交互模式，信息架构合理 |
| **Liquity** | 前端纯模拟的"预览"模式（不连接钱包也能查看），降低试用门槛 |

### 5.3 竞品分析洞察

1. **Web3 Dashboard 共同问题**：缺乏分层设计——所有用户看到完全相同的界面，没有根据用户类型/角色做差异化。
   - → **Contra 的机会**：4 种身份的差异化面板本身就是核心 UX 优势。

2. **治理投票 UX 普遍粗糙**：多数协议的投票入口深、流程长（4-5 步）、缺少投票后反馈。
   - → **Contra 的优势**：投票就在 Dashboard 首页，2-3 步完成，Modal 内闭环。

3. **收益展示过度依赖折线图**：多数 DeFi Dashboard 默认使用 TradingView 风格折线图，对非交易用户不友好。
   - → **PRD 已采纳环形进度条方案**，认知门槛更低。

4. **Web3 新用户的 Onboarding 缺口**：多数 Dashboard 假设用户已理解协议术语（TVL、APR、Staking）。
   - → **建议**：Contra Dashboard 应为每个数据指标提供 Tooltip 解释（ⓘ），尤其对 User 身份。

### 5.4 差异化 UX 策略总结

| 竞品常见做法 | Contra 差异化方案 |
|-------------|-------------------|
| 所有人同一界面 | 4 身份差异化面板 + 专属数据 |
| 收益用折线图 | 环形进度条 + 三列数字卡片（低认知负荷） |
| 投票入口深藏菜单 | 投票在 Dashboard 首页，Modal 闭环 |
| 缺乏身份状态反馈 | Mint 前后面板切换动画 + 身份色贯穿 |
| 技术术语无解释 | 每个指标带 ⓘ Tooltip，渐进教育用户 |
